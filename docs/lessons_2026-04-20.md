## DETAILED TECHNICAL LESSONS

### D1. ContactOut API — three specific behaviours to remember

1. **`/v1/stats` reports allocation/usage, NOT usability.** phone_quota minus phone_count returns a "credits unused" number that can be > 0 while work endpoints 403 blanket. Stats endpoints are billing metadata, not access gates. ALWAYS verify usability via the actual work endpoint with a real request.

2. **`/v1/people/enrich` bundles email + phone — no phone-only path exists.** When email credits = 0, entire endpoint 403s regardless of `fields=['phone']`, `include_phone=true`, or any variant. Phone credits are stranded behind the email credit gate. Probed verbatim today: `/v1/people/linkedin`, `/v1/people/profile` return HTML (not valid endpoints); `/v1/people/search + include_phone=true` returns 400.

3. **Credits can be consumed on 400-error calls (non-standard billing).** Our 82 calls with wrong param name `profile` (vs correct `linkedin_url`) returned 400 and may have consumed phone credits. Standard API billing consumes only on successful delivery. Refund question blocked by Dave's "don't email Sami."

Saved to memory: `feedback_stats_endpoint_inference.md`.

### D2. Leadmagic AU performance — calibrated numbers
- **AU mobile hit rate: 0-5%** (pilot 0/5 today, historical 0% per Directive #317 ceo_memory). Mobile is effectively dead for AU. Use ContactOut phone when unblocked, or accept phone-less delivery.
- **AU enterprise email hit rate: ~47%** on 171 real FM profiles. Not 80% as 5-profile pilot suggested. Pilot-yield variance can be massive at small N.
- **Billing:** hit-only — no charge on 404/miss. Safe to attempt liberally without burn risk.

### D3. Silent try/except ban applies to one-shot scripts too
F-DIAGNOSIS-01 Fix 2 (2026-04-14) ratified "silent failures are unacceptable for any provider" — extended to all waterfall providers. Today Elliot's one-shot FM enrichment script had try/except swallowing non-2xx responses. 82 ContactOut phone calls silently 400'd, 171 Leadmagic email calls silently 404'd. Zero errors surfaced. Credits burned, no data.

**Rule:** no production API call (pipeline or one-shot) may use try/except that swallows non-2xx without logging + raising. One-shot scripts are NOT exempt.

### D4. Pilot before production, always
Two live failures today prevented by pilots:
- Leadmagic mobile pilot (5 profiles) caught 0% AU coverage before full 171 × $0.077 = $13 burn
- Health-check pilot caught Leadmagic endpoint correctness before bulk run (via Dave's caveat)

Two live failures today caused by no pilot:
- ContactOut enrich wrong-param-name: 82 calls burned before silent failure surfaced
- Leadmagic email wrong-method-wrong-path: 171 calls returned 404s silently

**Rule:** every new or modified external-API script requires a 3-5 call pilot with verbatim response paste (LAW XIV) before bulk execution. Cost: 2-5 minutes + 3-5 credits. Saves hours and dollars.

### D5. Ad-hoc scripts must use configured parallelism
One-shot FM enrichment ran sequentially at 1.2s spacing (was going to take ~85 min). Pipeline config has `GLOBAL_SEM_LEADMAGIC=10` (10 concurrent Leadmagic calls) — ratified and ready. Kill + restart with sem=10 finished in ~30 seconds. 10× speedup.

Same pattern noted in 2026-04-15 commit `PR #323` ("the test script I just wrote inline doesn't USE it — it's a raw loop instead of using the configured semaphore").

**Rule:** before writing ANY external-API loop, check if a GLOBAL_SEM_* semaphore exists in `src/pipeline/pipeline_orchestrator.py` for that provider. If yes, use it. Sequential-only is the exception that requires justification.

### D6. Data durability — artefacts representing real spend must be committed immediately
`fm_raw_profiles.json` (4.8 MB, 3,043 ContactOut search credits of enrichment signal) existed ONLY on VPS disk after Stage 2. Not in git, not in Supabase, not backed up. One `rm` / disk corruption / `git clean -fdx` = total loss and credits can't be refunded.

Committed today after Dave prompted.

**Rule:** any pipeline artefact that represents paid API consumption gets committed to git AND persisted to Supabase (even if not technically required by the current directive scope). 30 seconds of work protects real dollars.

### D7. Schema propagation through pipeline stages
Stage 1 CSV had `parent_company`, `hq_state`, `is_anchor`. These dropped during Stage 2 → Stage 3 joins because downstream joined on `company_name` only. Final CSV rebuilt them at 61/81 fill rate (20 lost to company_name string variants like "ISS Facility Services Australia and New Zealand" vs "ISS Australia").

**Rule:** CSV schema must be verified against the deliverable spec at each stage handoff. Columns dropped mid-pipeline are either (a) explicitly out-of-scope for downstream stages or (b) a defect. Final-stage re-joins are a fallback, not a substitute for carrying columns through.

### D8. Filename honesty
Delivered `fm_prospects_100.csv` with 81 rows. Misleading. Renamed to `fm_prospects_81.csv` per peer review.

**Rule:** filenames must reflect actual contents, not target or plan. `fm_prospects_81.csv` is honest. `fm_prospects_100.csv` with 81 rows is optimistic labeling.

Aiden responded terminal-only to Dave on fresh session startup. Dave doesn't monitor terminal. Fix ratified earlier today: HARD BLOCK at session-startup step 2 (global) / step 4 (worktrees) requires `tg -g "test"` verification before first outbound. Propagated to:

### D10. PR close decisions require ground-truth verification
PR #331 was closed on "cherry-pick later if needed" reasoning. Aiden verified main's `domain_blocklist.py` was still at 289 lines — the 1,515-entry expansion was NOT on main. Closing would have lost 1,202 new blocklist entries + 13 enterprise categories + 6 confirmed D2 drops. Cherry-picked as commit `8a82becf` to preserve the data.
